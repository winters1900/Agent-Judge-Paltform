## 1. 项目目标与范围

**目标**：基于课程提供的外卖 Agent 评估数据集，自主实现若干评估指标，对 Agent 运行轨迹进行量化分析，并形成评估报告。

**范围**：
- 不实现外卖 Agent 本体；评估对象为给定的轨迹数据（`steps`/`expected_steps`/`final_answer`）。
- 不直接调用 Ragas 顶层评估 API；允许参考其指标思想并复现计算逻辑。

**产出**：
1. 项目分析文档（已在 `ragas_report/Ragas项目代码阅读报告.md`）。
2. 评估代码（至少 2 个指标实现）。
3. 评估报告（指标选择原因、实现过程、结果分析）。

## 2. 需求与约束

### 2.1 功能需求
- 读取并解析 `eval_dataset.json` 数据集。

  - **工具调用准确度**（Tool Call Accuracy）或变体。

## 3. 数据设计

### 3.1 输入数据格式
数据集为 JSON 数组，每个样本包含：
- `task_id`：任务唯一标识
- `user_query`：用户请求
- `ground_truth`：期望结果描述
- `steps`：Agent 实际执行轨迹
- `expected_steps`：期望正确执行轨迹
- `final_answer`：Agent 最终回复

`steps` 与 `expected_steps` 中每个元素包含：
- `step`：步骤序号
- `thought`：推理过程
- `tool_call`：工具名
- `input`：工具输入（JSON 字符串）
- `observation`：工具返回

### 3.2 中间结构
在代码中建议转为标准化对象：
- `ToolCall(name, args)`
- `Step(thought, tool_call, args, observation)`
- `Sample(task_id, user_query, ground_truth, steps, expected_steps, final_answer)`

## 4. 指标设计

至少实现 2 个指标，推荐如下组合：

### 指标 A：工具调用准确度（Tool Call Accuracy）
**目标**：评估 Agent 是否调用了正确工具、顺序是否一致、参数是否匹配。

**基本思路**：
1. 对齐 `steps` 和 `expected_steps` 的 `tool_call` 顺序。
2. 对每步参数做键值级匹配（可设权重）。
3. 得分 = 平均参数匹配分 × 覆盖率 × 顺序一致性因子。

### 指标 B：任务完成度（Task Completion）
**目标**：评估最终回复是否满足用户诉求。

**实现方式**（二选一）：
- **规则版**：根据 `ground_truth` 中关键动作词（下单、支付等）或订单号是否出现进行打分。
- **LLM 版**：构造 Prompt，让模型判断 `final_answer` 是否完成 `ground_truth`。

### 可选指标 C：规划合理性（Plan Quality）
**目标**：是否存在冗余步骤、缺失关键步骤、死循环。

**实现方式**：
- 统计重复工具调用；
- 关键步骤缺失则扣分；
- `steps` 长度与 `expected_steps` 差距过大则扣分。

## 5. 系统架构设计

### 5.1 模块划分
- `data/`：数据加载与解析
- `metrics/`：指标实现
- `evaluator/`：评估调度与聚合
- `report/`：结果导出与报告生成

### 5.2 逻辑流程
1. 读取 JSON 数据集。
2. 转为标准对象结构。
3. 对每个样本计算多个指标。
4. 汇总统计（均值、分布）。
5. 输出评估报告（Markdown）。

## 8. 风险与对策

- **数据噪声**：轨迹可能包含不完整或无效字段。→ 增加容错解析。
- **LLM 评估不稳定**：引入多次投票或设置置信度阈值。
- **指标主观性**：提供规则版指标作为对照基线。
