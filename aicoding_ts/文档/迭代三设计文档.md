# AI Coding Agent Web 迭代三设计文档

## 1. 迭代目标

迭代三的核心目标是在迭代二 MVP 基础上完成两件事：

1. **Agent 产品化** — 将 Agent 应用从一个"能用的工具"升级为"完整的领域产品"
2. **Agent 能力优化** — 从评估指标和 Bad Case 出发，系统性地提升 Agent 效果

迭代时间：5-6 周，截止 **2026-06-15 23:59**。

---

## 2. 迭代二成果回顾

迭代二已完成以下能力，是迭代三的起点：

| 模块 | 已实现 | 关键文件 |
|---|---|---|
| Agent 编排 | Planner → Executor(ReAct) → Reviewer → Summarizer 四角色流程 | `packages/agent-core/` |
| 工具网关 | 9 个本地工具 + MCP 协议支持 + 外部 MCP 集成 | `packages/tool-gateway/`, `packages/mcp-server/`, `packages/mcp-client/` |
| 上下文构建 | 基于关键词匹配的文件评分排序 + token 预算控制 | `packages/context-builder/` |
| 工作区管理 | 内存文件树 + 磁盘同步 + unified diff patch + 快照/回滚 | `packages/workspace-manager/` |
| 会话存储 | 文件持久化会话 + 消息历史 + 任务摘要 | `packages/session-store/` |

| 运行时服务 | HTTP + SSE + MCP JSON-RPC 端点 | `apps/runtime/server.ts` |

### 迭代二中待解决的问题

1. **多轮对话能力弱** — 消息历史仅做简单截断（最近 40 条），无智能压缩和记忆管理
2. **上下文构建粗糙** — 仅靠关键词 token 匹配评分，未利用语义检索
3. **用户无法管理对话** — 会话列表可查看但无法删除/归档/导出
4. **用户无法管理工具** — 外部 MCP 工具只能在 `mcp-config.ts` 页面配置，无统一管理入口
5. **无 Skill 系统** — Agent 的能力完全由 system prompt 和工具列表决定，无法动态加载领域能力
6. **无跨对话知识共享** — 每个会话独立，知识和经验无法复用
7. **评估和优化闭环缺失** — 没有评估数据集，无法量化优化效果

## 3. 任务一：Agent 产品化

### 3.1 基本需求：对话与工具管理

#### 3.1.1 对话管理

### 3.2 加分需求一：跨对话知识共享

**目标**：用户可以让 Agent 在不同会话之间共享和复用知识。

#### 3.2.1 设计思路

当前系统每个会话完全独立，Agent 在会话 A 中学到的经验无法用于会话 B。跨对话知识共享的核心是建立一个**项目级知识库**，Agent 可以在执行任务时按需检索。

#### 3.2.2 知识类型

| 类型 | 内容 | 示例 |
|---|---|---|
| 项目规范 | 编码规范、命名约定、架构决策 | "本项目使用 Prettier 格式化，缩进 2 空格" |
| 技术决策 | 为什么选择某个库/方案 | "选择 zod 而非 yup 是因为更好的 TS 推导" |
| 常见问题 | FAQ 和解决方案 | "本项目的认证 token 需要从 /api/auth 获取" |
| 用户偏好 | 用户的编码风格和习惯 | "用户偏好函数式组件，不使用 class component" |
| 任务模式 | 常用任务的处理模式 | "新增 API 端点需要同时添加路由、service、test" |

#### 3.2.3 架构设计

```
┌────────────────────────────────┐
│       Project Memory Store     │
│  项目级知识库（跨会话共享）       │
│  - 手动录入                     │
│  - Agent 自动提取               │
│  - 向量化存储（可选）            │
└───────────┬────────────────────┘
            │
    ┌───────┴────────┐
    │                │
┌───▼─────┐   ┌──────▼──────┐
│ 会话 A   │   │  会话 B      │
│ 知识注入  │   │  知识检索    │
└─────────┘   └─────────────┘
```

#### 3.2.4 实现方案

**存储**：扩展 `workspaces/<projectId>/project-memory.md`（`session-store` 已预留此文件）：

```markdown
# 项目记忆

## 编码规范
- 使用 TypeScript 严格模式
- 缩进 2 空格，不使用 tab

## 技术栈
- 前端：React 18 + Vite
- 后端：Express + TypeScript
- 校验：zod

## 已知问题
- Windows 环境下 pnpm 安装需要 --shamefully-hoist

## Agent 任务经验
- 2026-05-20: 修改 API 路由时需同步更新 types.ts
- 2026-05-22: npm install 后需要重启 dev server
```

**检索方式**：

| 方式 | 适用场景 | 实现 |
|---|---|---|
| 全量注入 | 知识量少（< 2KB） | 直接附加到 system prompt |
| 关键词匹配 | 任务明确对应某类知识 | 对 prompt 做 token 化后匹配知识条目 |
| 向量检索（可选） | 知识量大，跨话题 | embedding + 余弦相似度 |

**UI 入口**：

- 在工作区面板增加"项目知识"Tab
- 支持手动编辑 Markdown 格式的项目记忆
- Agent 在任务摘要中可建议"将此经验保存到项目知识"
- 用户可选择是否将当前会话的某条经验标记为"跨会话共享"

#### 3.2.5 Context Builder 集成

在 `packages/context-builder/index.ts` 中，`buildForPrompt` 新增知识检索步骤：

```
1. 读取 project-memory.md
2. 根据 prompt 关键词匹配相关段落
3. 将匹配结果加入上下文中的"项目知识"段
4. 计入 token 预算
```

### 3.3 加分需求二：Skill 系统

**目标**：用户可以向 Agent 中添加 Skill，动态扩展其领域能力。

#### 3.3.1 什么是 Skill

Skill 是一个可复用的能力包，包含：

- **触发条件**：什么情况下激活此 Skill
- **领域知识**：该领域的专业 prompt 指令
- **工具扩展**：该领域专用的工具或 MCP 集成
- **工作流**：该领域特定的任务处理流程

参考：[Anthropic Skills](https://github.com/anthropics/skills/tree/main/skills)，[Skill Creator](https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md)

#### 3.3.2 Skill 定义格式

```typescript
// packages/skill-system/types.ts
type SkillDefinition = {
  name: string;                    // 唯一标识
  displayName: string;             // 显示名称
  description: string;             // 简短描述
  version: string;
  
  // 触发条件
  triggers: {
    keywords?: string[];           // 关键词触发
    filePatterns?: string[];       // 文件类型触发（如 "*.sql"）
    always?: boolean;              // 始终激活
  };
  
  // 能力注入
  systemPromptExtension: string;   // 追加到 system prompt 的内容
  tools?: SkillTool[];            // 此 Skill 引入的专用工具
  mcpServers?: ExternalMcpServerConfig[]; // 此 Skill 需要的 MCP 服务
  
  // 工作流（可选）
  workflow?: {
    steps: SkillStep[];           // 预定义的任务步骤
  };
};

type SkillTool = {
  name: string;
  description: string;
  inputSchema: JsonSchema;
  handler: string;                // 工具处理逻辑（代码字符串或函数引用）
};
```

#### 3.3.3 架构设计

```
┌─────────────────────────────────────┐
│           Skill Registry             │
│  管理所有已安装的 Skill               │
│  - 注册 / 卸载                       │
│  - 触发匹配                          │
│  - 版本管理                          │
└──────────────┬──────────────────────┘
               │
    ┌──────────┼──────────┐
    │          │          │
┌───▼───┐ ┌───▼───┐ ┌───▼───┐
│Skill A│ │Skill B│ │Skill C│
│ 前端  │ │ 数据库 │ │ 测试  │
└───────┘ └───────┘ └───────┘
```

#### 3.3.4 核心流程

```
用户输入请求
    │
    ▼
Agent Orchestrator 接收请求
    │
    ▼
Skill Registry 匹配触发条件
    ├── 关键词匹配
    ├── 文件类型匹配
    └── 始终激活
    │
    ▼
激活匹配的 Skill(s)
    ├── 注入 system prompt 扩展
    ├── 注册 Skill 专用工具
    └── 应用 Skill 工作流
    │
    ▼
执行任务（Executor → Reviewer → Summarizer）
```

## 4. 任务二：Agent 能力优化

### 4.1 基本需求：多轮对话支持

**目标**：Agent 能在一次多轮对话中正确理解上下文，不丢失关键信息。

#### 4.1.1 当前问题

| 问题 | 现状 | 影响 |
|---|---|---|
| 简单截断 | 消息超过 40 条直接丢弃早期消息 | 丢失关键上下文 |
| 无摘要压缩 | 历史消息全部保留原文 | token 膨胀，成本高 |
| 无对话窗口 | 不支持滑动窗口管理 | 长对话退化严重 |
| 任务上下文断裂 | 前后任务无关联信息传递 | 重复询问和错误 |

#### 4.1.2 优化方案

**方案一：滑动窗口 + 关键消息保留**

```
┌─────────────────────────────────────────┐
│        完整消息历史（无限制）              │
│  [msg1] [msg2] [msg3] ... [msg40] [msg41]│
└────────────────┬────────────────────────┘
                 │ 策略筛选
                 ▼
┌─────────────────────────────────────────┐
│        送入 LLM 的消息（有限窗口）         │
│  [system] + [关键msg] + [最近N条]         │
└─────────────────────────────────────────┘
```

规则：
1. System prompt 和任务摘要**始终保留**
2. 包含 `write_file`、`patch_file`、`run_command` 的轮次**优先保留**（含工具调用结果）
3. 纯文本对话轮次可被摘要压缩
4. 最近 5 轮对话**完整保留**
5. 总 token 控制在模型上下文窗口的 80% 以内

**方案二：分层记忆机制**

借鉴认知心理学中的记忆模型：

```
┌─────────────┐
│  工作记忆    │ ← 当前任务上下文（最近 N 轮对话 + 当前打开文件）
│  (Working)  │   容量小，更新快
├─────────────┤
│  短期记忆    │ ← 当前会话历史（摘要后的历史消息）
│  (Short)    │   容量中，会话级别
├─────────────┤
│  长期记忆    │ ← 跨会话知识（项目记忆 + 任务摘要 + 用户偏好）
│  (Long)     │   容量大，项目级别
└─────────────┘
```

实现在 `packages/context-builder/` 中新增 `MemoryManager`：

```typescript
// packages/context-builder/memory-manager.ts
type MemoryManager = {
  // 工作记忆：组装当前轮次上下文
  buildWorkingMemory(prompt: string, recentMessages: ChatMessage[]): Context;
  
  // 短期记忆：压缩历史消息
  compressHistory(messages: ChatMessage[], maxTokens: number): string;
  
  // 长期记忆：从项目知识库检索
  retrieveLongTermMemory(prompt: string): Promise<string>;
};
```

**短期记忆压缩策略**：

对历史消息按轮次分组，每组（用户消息 + Agent 响应 + 工具调用结果）生成一句摘要：

```
原始（可能 2000 tokens）：
User: 帮我创建一个 Express API 项目
Agent: [调用 write_file × 5, run_command × 2]...
Agent: 项目已创建，包含以下文件...

压缩后（约 50 tokens）：
[第1轮] 用户请求创建 Express API 项目 → Agent 创建了 5 个文件并执行 npm install
```

使用 LLM 调用进行摘要（用便宜的模型，或复用当前模型的一次低成本调用）。

**多轮对话增强 prompt 模板**：

```
## 会话摘要
- 本轮是此会话的第 5 轮对话
- 前 4 轮完成了：Express API 项目创建、添加了用户认证路由、修复了中间件 bug
- 当前工作区包含 12 个文件，最近修改的是 src/routes/auth.ts

## 近期操作
- write_file: src/routes/auth.ts (+45 行)
- run_command: npm run build (exit 0)
```

### 4.2 亮点扩展：评估驱动的 Agent 优化

#### 4.2.1 建立评估数据集

**评估数据集结构**：

```typescript
type EvalCase = {
  id: string;
  category: 'scaffold' | 'modify' | 'fix' | 'explain' | 'refactor';
  difficulty: 'easy' | 'medium' | 'hard';
  prompt: string;                  // 用户输入
  workspaceState: {                // 初始工作区状态
    files: Array<{ path: string; content: string }>;
  };
  expectedCriteria: {              // 评判标准（非精确匹配）
    fileChanges?: Array<{ path: string; shouldContain?: string[]; shouldNotContain?: string[] }>;
    commandResults?: Array<{ command: string; expectExitCode: number }>;
    responseContains?: string[];   // Agent 回复中应包含的内容
    responseNotContains?: string[]; // Agent 回复中不应包含的内容
  };
};
```

**数据集构建方式**：
1. 手动编写 20-30 个典型用例（覆盖不同难度和类别）
2. 从迭代二使用过程中收集的真实 Bad Case
3. 模板生成场景的专项测试用例

**评估流程**：

```
评估数据集
    │
    ▼
逐条运行 eval case
    │
    ▼
Agent 执行任务（在隔离的临时工作区）
    │
    ▼
自动评判（规则检查 + LLM 评判）
    ├── 文件变更是否正确
    ├── 命令执行是否符合预期
    └── 回复内容是否满足要求
    │
    ▼
汇总指标：成功率、平均轮次、工具调用次数、token 消耗
```

#### 4.2.2 四个优化方向

##### 方向一：Agent 架构优化

**1. 混合推理模式**

当前 Executor 的 ReAct 循环是纯自治模式（最高 20 轮迭代，每轮 LLM 决定下一步）。

优化方案：引入 **Plan-then-Execute** 模式用于确定性强的任务：

```
当前模式（ReAct）：
  Think → Act → Observe → Think → Act → Observe → ...

优化模式（混合）：
  简单任务（如"读取文件 X"）→ 直接执行，跳过规划
  中等任务（如"修改文件 Y 的某个函数"）→ Plan → Execute → Review
  复杂任务（如"创建一个完整的登录系统"）→ Plan → Execute(多步) → Review → Summarize
```

实现：在 `agent-core/index.ts` 的 `runTask()` 中增加任务复杂度判断：

```typescript
function assessComplexity(prompt: string): 'simple' | 'medium' | 'complex' {
  // 基于 prompt 长度、关键词、涉及文件数量等因素判断
}
```

**2. Multi-Agent 探索（可选）**

对于涉及截然不同专业能力的任务，考虑使用多个专业化 Agent：

- **Code Agent**：负责代码编写和修改
- **Review Agent**：负责代码审查和质量检查
- **Doc Agent**：负责文档生成和维护

但迭代三中谨慎引入，避免过度工程化。仅在评估数据显示单一 Agent 在特定跨领域任务上表现不佳时才考虑。

**3. Workflow 与自治循环的灵活切换**

- 明确性高的任务（如模板生成）：使用固定的 workflow
- 探索性任务（如"帮我改进这个项目的性能"）：使用自治 ReAct 循环
- 混合任务：workflow 框架 + 关键节点 LLM 决策

##### 方向二：上下文工程优化

**优化项**：

| 优化 | 现状 | 目标 |
|---|---|---|
| 文件评分算法 | 纯关键词 token 匹配 | 加入语义相似度（可选 embedding） |
| 长文件处理 | 简单截断首尾 | 按函数/类切片，只取相关部分 |
| 上下文去重 | 无 | 检测重复内容并去重 |
| 上下文排序 | 按评分降序 | 按"重要性+相关性"排序，关键文件靠前 |
| Token 预算动态分配 | 固定 maxFiles=8, maxChars=12000 | 根据任务复杂度动态调整 |

**代码改动**：主要改动在 `packages/context-builder/index.ts`：

```typescript
// 增强的评分函数
function scoreFileV2(filePath: string, fileContent: string, taskContext: TaskContext): number {
  let score = 0;
  
  // 关键词匹配（保留）
  score += keywordMatchScore(filePath, taskContext.promptTokens);
  
  // 文件类型权重
  score += fileTypeWeight(filePath);  // 入口文件、配置文件加权
  
  // 最近修改加权
  score += recencyWeight(filePath, taskContext.recentChanges);
  
  // 引用关系加权
  score += referenceWeight(filePath, taskContext.currentFile);
  
  // 语义相似度（可选，在项目较大时启用）
  if (taskContext.projectSize === 'large') {
    score += semanticSimilarityScore(filePath, taskContext.prompt);
  }
  
  return score;
}
```

##### 方向三：RAG 知识增强

**适用场景判断**：

RAG 不是默认开启，而是在以下条件下触发：
- 项目文件数 > 50
- 关键词匹配得到的文件相关性分数普遍较低（< 阈值）
- 用户 prompt 比较抽象，不直接指向具体文件

**实现方案**：

```
文件变更时
    │
    ▼
增量更新向量索引（可选，嵌入模型）
    │
    ▼
任务到来时
    │
    ▼
┌─────────────────────────────┐
│ 第一层：启发式检索（快速）    │
│ - 关键词匹配 + 文件评分      │
│ - 适用于 80% 的常见场景      │
└─────────────┬───────────────┘
              │ 分数不足
              ▼
┌─────────────────────────────┐
│ 第二层：向量检索（兜底）      │
│ - Embedding 相似度搜索       │
│ - 适用于大项目 / 模糊需求     │
└─────────────────────────────┘
```

**向量化方案选择**：

| 方案 | 优点 | 缺点 |
|---|---|---|
| 本地 embedding（如 transformers.js） | 无外部依赖，隐私好 | 模型体积大，首次加载慢 |
| API embedding（如 OpenAI/Doubao） | 速度快，质量高 | 需要 API key，有成本 |
| 关键词-only（不引入向量） | 简单，零成本 | 语义理解弱 |

MVP 阶段建议先用**关键词 + 启发式**（当前已有），在评估数据显示明显不足时再引入向量检索。如果引入，优先使用已有的 LLM API 做 embedding（如 Doubao embedding API）。

**知识标注出处**：

Agent 回答中引用文件内容时，标注来源文件和行号：

```
✅ 根据 src/utils/auth.ts:42-58，当前的 token 验证逻辑在中间件中...
```

在 system prompt 中增加指令：引用文件内容时需标注路径。

##### 方向四：工具链重构与扩展

**优化项**：

| 优化 | 现状 | 目标 |

| `patch_file` 容错 | 简单的 before/after 匹配 | 增加 fuzzy match、上下文行号定位、多格式兼容 |

| 新增 `read_lints` 工具 | 无 | 读取工作区的 lint 错误 |

## 7. 风险与应对

| 风险 | 概率 | 影响 | 应对 |

## 8. 总结

迭代三的核心思路是**从"能用"到"好用"**：

- **产品化**：让用户能管理对话和工具，通过 Skill 系统和跨会话知识共享让 Agent 能力可扩展
- **能力优化**：从多轮对话、上下文工程、工具链安全三个维度系统性地提升 Agent 的任务成功率

技术实现上以**渐进增强**为原则：优先完成 P0 的基本需求，P1 补全关键能力，P2 的亮点功能视时间灵活调整。所有优化都必须基于评估数据，避免盲目改动。
