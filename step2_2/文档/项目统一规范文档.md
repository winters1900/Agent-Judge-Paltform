# 项目统一规范文档

本文档用于统一 `step2_2` 项目中通用 Agent 评估平台的术语、字段命名、状态枚举、资源边界与接口约定。后续需求文档、API 文档、数据库文档、后端代码与前端页面都应以此为准。

---

## 1. 设计目标

本项目目标是构建一个**通用 Agent 评估平台**，支持对不同类型、不同接入方式、不同能力侧重的 Agent 应用进行统一评测。平台应支持：

- 评测目标管理
- 评测任务管理
- 评测运行管理
- 数据集与样本管理
- 评估方法管理
- 指标定义与结果管理
- 过程轨迹与工具调用记录
- 评测报告导出
- 多任务对比分析

Coding 场景仅作为示例场景之一，不作为平台能力边界。

## 2. 核心资源定义

### 2.1 评测目标 `EvaluationTarget`
被测对象的统一抽象。

可表示为：

- HTTP Agent
- CLI Agent
- Workflow Agent
- Retrieval Agent
- Multi-turn Agent
- 自定义适配系统

### 2.2 评测任务 `EvaluationTask`
评测配置与执行计划的统一抽象。

任务负责绑定：

- 评测目标
- 数据集
- 评估方法
- 指标配置
- 运行参数

### 2.3 评测运行 `EvaluationRun`
一次评测任务的实际执行实例。

一个任务可以对应多个运行实例。

### 2.4 数据集 `Dataset`
评测样本的集合。

### 2.5 数据集样本 `DatasetSample`
单条评测输入。

### 2.6 评估方法 `EvaluationMethod`
定义“怎么评”的方法或策略。

### 2.7 指标定义 `MetricDefinition`
定义“评什么”的指标。

### 2.8 指标结果 `MetricResult`
某次运行或某个样本的指标输出结果。

### 2.9 过程轨迹 `EvaluationTrace`
记录 Agent 执行过程中的步骤、决策、观察与状态快照。

### 2.10 工具调用日志 `ToolCallLog`
记录 Agent 对外部工具的调用详情。

### 2.11 样本结果 `EvaluationSampleResult`
记录样本级执行结果、输入输出快照与评分摘要。

### 2.12 评测报告 `EvaluationReport`
记录单次运行的报告信息与导出路径。

### 2.13 对比分析 `AnalysisResult`
记录多任务或多指标的横向比较结果。

## 3. 统一字段命名规范

### 3.1 通用命名原则

1. 优先使用平台统一术语
2. 同类字段跨文档、跨接口、跨表保持一致
3. 不同层次的配置与执行结果分开命名
4. 统一使用 `metadata`，不要混用 `sample_metadata`
5. 统一使用 `target` 表示被测对象，不再把 `agent_id` 当成唯一主概念

## 6. 接口与数据库映射原则

### 6.1 资源映射

| 资源 | API 路径 | 数据表 |
|---|---|---|
| 评测目标 | `/api/v1/evaluation-targets` | `evaluation_target` |
| 评测任务 | `/api/v1/evaluation-tasks` | `evaluation_task` |
| 评测运行 | `/api/v1/evaluation-runs` | `evaluation_run` |
| 数据集 | `/api/v1/datasets` | `dataset` |
| 数据集样本 | `/api/v1/datasets/{dataset_id}/samples` | `dataset_sample` |
| 评估方法 | `/api/v1/evaluation-methods` | `evaluation_method` |
| 指标定义 | `/api/v1/metrics` | `metric_definition` |
| 指标结果 | `/api/v1/evaluation-runs/{run_id}/metrics` | `metric_result` |
| 过程轨迹 | `/api/v1/evaluation-runs/{run_id}/traces` | `evaluation_trace` |
| 工具调用日志 | `/api/v1/evaluation-runs/{run_id}/tool-calls` | `tool_call_log` |
| 报告 | `/api/v1/evaluation-runs/{run_id}/export` | `evaluation_report` |
| 对比分析 | `/api/v1/analysis/compare` | `analysis_result` |

### 6.2 核心规则

1. API 中的对象名称应与文档术语保持一致
2. 数据库字段应与 API 请求/响应中的字段尽量一致
3. 配置字段与执行结果字段分开保存
4. 冗余快照字段用于历史追溯，不用于主数据修改
5. 运行状态与任务状态分开定义
6. `metadata` 为统一扩展字段名
7. `target` 表示被测对象，不再以 `agent_id` 作为平台唯一主概念

## 7. 文档编写与代码实现要求

### 7.1 文档要求
- 三份文档中的术语必须一致
- 字段名、接口路径、枚举值必须统一
- Coding 场景只能作为示例，不得覆盖通用场景
- 评测目标必须作为平台一级资源独立存在

### 7.2 代码实现要求
- 后端模型、Schema、Repository、Service、API 命名应统一
- 前端表单字段名应与 API 字段名一致
- 数据库字段名应与后端模型属性尽量一致
- 状态机逻辑应严格区分任务状态与运行状态
